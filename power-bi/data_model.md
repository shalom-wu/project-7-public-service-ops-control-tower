## Relationships to create

| From | To | Cardinality | Notes |
|---|---|---|---|
| `fact_service_requests[created_date_only]` | `dim_date[date]` | many-to-one, single direction | create `created_date_only = DATEVALUE(fact[created_date])` as a Power Query step or calculated column, then relate |
| *(none for `dim_sla_targets`)* | | | SLA fields (`sla_days`, `sla_status`) are already joined into the fact by the SQL layer; keep `dim_sla_targets` as a disconnected reference table so users can inspect the assumed targets |
| *(none for `priority_scores`)* | | | disconnected on purpose: it is scored at complaint-type × borough grain and drives the Priority page on its own |

Mark `dim_date` as the model's **date table** (Table tools → Mark as date
table) so time intelligence (`PREVIOUSMONTH`, etc.) works.

## What-if parameters (Priority & Action page)

Create four numeric-range parameters (Modeling → New parameter):

| Parameter | Range | Step | Default |
|---|---|---|---|
| `W Volume` | 0 – 1 | 0.05 | 0.30 |
| `W Backlog` | 0 – 1 | 0.05 | 0.25 |
| `W Miss` | 0 – 1 | 0.05 | 0.25 |
| `W Aged` | 0 – 1 | 0.05 | 0.20 |

`priority_scores` carries `volume_percentile` and `backlog_percentile`
pre-computed by SQL precisely so the dynamic score is simple DAX
(see `dax_measures.md → Dynamic Priority Score`).

## Why the fact table ships as Parquet

1.07M rows: as CSV it is ~227MB (unfit for GitHub); as ZSTD Parquet it is
~14MB and Power BI Desktop loads it natively (Get Data → Parquet).
